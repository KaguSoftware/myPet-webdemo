# PetPal — project context for AI assistants

Read this before working on the project. It captures scope, decisions, and how to ship.

**Keep this file current.** As you make meaningful changes — new features, design-direction shifts, architectural moves, new gotchas, things you tested — update the relevant section here in the same working session so the next AI chat inherits an accurate picture. This file is the running memory of the project; stale context here is worse than none.

## What this is
**PetPal** is a **phone-only web app** for managing a family's pet care. It began as a clickable demo and is now a **real product**: the UI is polished enough to later "convert to app code" with minimal changes, and it is backed by a live database with real accounts.

Tech: **Next.js 16.2+ (App Router, TypeScript strict, Tailwind v4), deployed on Vercel.** Persistence and auth are **Supabase** — email/password auth (`@supabase/ssr`, cookie sessions), Postgres with **Row Level Security** scoped by `auth.uid()`, and SQL migrations in `supabase/migrations/`. Every route is behind a `/login` wall enforced by `proxy.ts` (the Next middleware). There is **no localStorage** persistence — the React store hydrates from and writes to Supabase.

## Product concept
A family pet-care hub with two tiers:
- **Free tier**: family members log care actions (fed, water, litter, walk, groomed, meds, vet); everyone is "notified" (in-app feed + toasts, simulated from the shared activity history). Manual reminders included.
- **Premium ("PetPal+")**: vet-built, breed-specific care plans; smart reminders; sponsored vet-booking suggestions. Upgrading is a fake instant unlock from the Family tab — no payment. When premium is on, the Home dashboard also surfaces the day's care-plan checklist.
- **Gamification**: coins earned per logged action, streaks (derived from real activity history), and a cosmetics/dress-up system to put hats/glasses/collars/outfits on the pets. (The XP/level system was removed — it didn't tie into anything else in the product. `state.xp`/`level()`/`LevelStagesSheet` are gone; the DB `households.xp` column is still there but unused by the app.)

Multiple family members share one **household** row; switching member is a "view as" — actions are logged as the selected member. New signups get a household seeded by the `handle_new_user` DB trigger.

## Design direction (important — this has iterated)
The look was deliberately tuned on a "fun scale." History: v1 = 100% fun (pastel/emoji/confetti, read as a cheap toy) → rebuilt to clean **iOS "Liquid Glass" native** = 0% fun (sterile) → warmed to ~40% → ~60-70% fun via a full pixel-art theme (pixel icons, pixel page titles, pixel tab labels) → **Jul 2026 "real app" pass: the pixel language is now confined to the pets and their world; all chrome is clean native iOS.** The owner's complaint was "doesn't feel like a real app" — the diagnosis was two visual languages fighting (arcade font/icons in navigation, headers, settings — exactly where real products stay neutral). The model is Duolingo: the *character* is illustrated, the *chrome* is native.

Current state (the "pixel pets, native chrome" pass):
- **Where pixel is ALLOWED**: the pets themselves (`PixelPet`/`PetAvatar`/`Pet3D`), cosmetics sprites, the coin sprite in `CoinPill`, the `.arcade-stage` grid behind the dress-up stage and Welcome step 0, and the pixel cat as the auth-screen brand mark (`components/BrandMark.tsx` — mascot over a clean Inter wordmark). Arcade motion stays (`idle` wobble, `+5` coin-pop, coin-bump, buy/equip pop).
- **Where pixel is BANNED**: everything else. Icons, page titles, tab labels, toasts, charts, member avatars, auth type — all clean iOS (Inter / stroke icons). Do not reintroduce pixel styling into chrome.
- **NO EMOJI IN THE UI — ever.** The owner explicitly hates emoji. Emoji still exists as *data* (DB `emoji` columns on reminders/members/pets/cosmetics, `CARE_PLANS` item keys, `CARE_WARNING_EMOJI` lookups) but must never render. Everything visual goes through `Icon` / sprites. Toasts take an `IconName`, not an emoji (see below).
- **Icons are clean stroke SVG** (SF Symbols / Lucide register). `components/Icons.tsx`: each icon is JSX path data in a 24×24 viewBox, `stroke=currentColor`, strokeWidth 2, round caps/joins (small dots use per-element `fill`). Public API unchanged: `name` / `size` / `className`. Extra names added along the way: `alert`, `trash`, `scale`, and (health/scheduling pass) `syringe`, `repeat`, `share`, `gift`. To add an icon, author paths in the same register — no pixel grids, no filled blob icons.
- **Toast API**: `toast(icon: IconName, title, body?, action?)` — the store's `Toast` type carries `icon` plus an optional `action: { label, onClick }` rendered as a button (tapping it also dismisses the toast); `Toasts.tsx` derives the tile tone from the icon (`alert`/`trash` → red, `check`/`star`/`flame` → green, else indigo). When adding a toast call, pick a semantic icon name. Every `logAction` success toast carries an Undo action (→ `undoLogAction`).
- **Pixel-art pets**: pets render as hand-authored pixel sprites. Engine is `components/pixel/` — `PixelSprite.tsx` (renders a sprite from string-rows + a char→color palette as `<rect>` blocks, run-length merged, `shape-rendering: crispEdges`; `size` sets width, height scales by the row/col ratio), `petSprites.ts` (16×16 cat/dog faces), `petBodySprites.ts` (16×24 full bodies for the 3D viewer), `cosmeticSprites.ts` (accessories as pixel sprites with per-slot placement fractions), `hudSprites.ts` (coin/heart/star glyphs), and `PixelPet.tsx` which composes the pet + layers equipped cosmetics on top (exports `PixelCosmetic` for shop previews and `equippedCosmetics(pet)` — the shared list used by both `PixelPet` and `Pet3D`).
- **`PetAvatar.tsx`** wraps `PixelPet` as a **background gradient disc + an unclipped pet layer** (no `overflow-hidden`), so equipped **hats poke above the round frame** (and capes/outfits slightly below) exactly as they appear on the full-size dress-up stage — preview == reality. Same `size` (xs/sm/md/lg/xl) + `idle` + `showCosmetics` API. `InitialAvatar` is now a **clean gradient circle with an Inter initial** (people are not part of the pixel pet world).
- **Pixel cosmetics**: accessories are pixel sprites on the pet, the head `+` button, and the shop grid. `data.ts` still stores `emoji` per cosmetic as a fallback (`PixelCosmetic` falls back to emoji if a sprite id is missing — every current cosmetic has a sprite, keep it that way). The `Pet3D` viewer (dress-up "3D" toggle) also renders equipped cosmetics, re-anchored onto the body sprite's head box.
- **Weight chart** (`components/pixel/PixelChart.tsx` — name is legacy): now a standard smooth trend line with round caps, circle data points, green target band, Inter numbers. Don't re-pixel it.
- **Quieter gamification**: coins/streak stay but are visited, not stared at. `CoinPill` (pixel coin sprite + Inter number) appears **only in the shop sheets** on `/pets`. The day-streak's surface is **Settings ▸ Account** ("Day streak" row → `StreakCalendarSheet`) — it was removed from Home's header. The `+5` coin-pop on `/logs` (Inter bold now) and the streak-up toast remain as logging feedback.
- **Hue discipline**: the accent is oklch hue **285**. Several shadows/gradients were hardcoded at 258/262 and looked subtly off-brand — all corrected to 285-292. Never hardcode a 258-hue indigo again.
- **Pixel font (Geist Pixel)**: still wired up (`app/fonts/GeistPixel.ttf` via `next/font/local`, `--font-pixel` / `.font-pixel`) but **currently unused** after this pass — kept only in case a pet-world surface wants it later. Do not use it for chrome. (NOT next/font/google — Geist Pixel is not in this pinned Next version's font-data list, so importing it there errors at build. The TTF is the variable font instanced at `ELSH=0`.)
  - **Gotcha that bit us repeatedly — how `.font-pixel` wires up:** `next/font/local` injects `--font-pixel` as a class on `<body>`, so that var only exists in `<body>`'s cascade, NOT on `:root`. Do **not** build a font stack inside Tailwind's `@theme` (e.g. `--font-pixel-stack: var(--font-pixel), …`) — `@theme` emits at `:root`, where `--font-pixel` is undefined, so the whole `var()` collapses to empty and the pixel font silently never applies (elements fall back to Tailwind's `font-sans`). The `.font-pixel` rule in `globals.css` must reference `var(--font-pixel), "Courier New", monospace` **directly** so it inherits from `<body>`. If a previous session's TTF ever rendered as a plain serif/sans, suspect either (a) this `:root` var trap, or (b) a bad/mislabeled TTF — verify by rendering the file in isolation.
  - **Re-fetching the font**: the genuine variable font is `ofl/geistpixel/GeistPixel[ELSH].ttf` in the google/fonts GitHub repo (~3.6MB). Instance it to a small static TTF with `fontTools.varLib.instancer.instantiateVariableFont(font, {'ELSH': 0})`. Google's `css2` static export renders as a non-pixel serif-ish default — don't use it.
- **Arcade motion is ON** (pet-world only): `idle` sprite wobble on hero/dress-up pets, `+5` coin-pop on logging a care action, `CoinPill` bump on any coin gain, dress-up buy/equip pop. Keyframes live in `globals.css` (`idle`, `coin-pop`, `coin-bump`, `pop`, `toast-in`); all respect `prefers-reduced-motion`. The dress-up stage has an `.arcade-stage` retro-grid backdrop.
- Copy stays professional (no jokey microcopy) unless the user asks.

To author a new sprite: add rows (equal-length strings) + palette to the relevant `*Sprites.ts` file; transparent = `.` or space. Cosmetics need a `place` (left/top/widthFrac as fractions of the 16px pet box). Keep pets on a 16×16 grid so cosmetic placement stays aligned. **Rows must be equal length** — `PixelSprite` reads `rows[0].length` for width, so a longer row gets clipped.

If asked to change the "fun level" again, it's a dial on: how far the pet-world (sprites, arcade motion, stage backdrop) is allowed to spill into the chrome — not a rewrite. The current setting is "pets only".

## Architecture / where things live
- `lib/data.ts` — types + **static reference data** (cosmetics catalogue, breed care plans, `VETS` marketplace list with `sponsored` flags; `VET` = `VETS[0]` for dashboard cards) + pure helpers: `formatWeight`/`kgToUnit`/`unitToKg`/`weightUnitLabel` (share `LB_PER_KG`), `dailyTarget(pet, type)`/`dailyGramTarget(pet)`, `formatAge`, `isAdminRole`/`ADMIN_ROLE`. The `AppState`/`Pet`/`Member`/`Activity`/`Reminder` types live here. (There is **no** `SEED` sample state any more — real data comes from Supabase; the on-signup seed lives in the DB trigger.)
  - **Health & scheduling types (Jul-2026 health pass):** `Vaccination` (name, `dateGiven`, optional `nextDue`, notes) and `VetVisit` (ts, vetName, reason, notes) live on `Pet` as `vaccinations`/`vetVisits`. `Pet` also gained identity fields: `createdAt` (gotcha day), optional `birthDate` (ms epoch — when set, `ageYears` is DERIVED at hydration and the age editors switch to editing the date; typing a manual age on Home clears `birthDate` back to approximate-age mode), `microchip`, `allergies`, `notes`. `WeightPoint` now carries `id` (weights are deletable). `Reminder` gained `repeatKind` (`RepeatKind` = daily/weekly/every_n_days), `repeatInterval`, and `vaccinationId` (links auto vaccine reminders; DB-cascaded). Helpers: `nextRepeatDue(due, kind, interval?)` (rolls forward past now preserving time-of-day), `ageYearsFromBirthDate`, `nextBirthday`, `nextAnniversary`.
  - **Age/gender-aware breed guidance (added when `CARE_PLANS` was rewritten from the vet reference doc):** `WEIGHT_FEEDING_GUIDES[breed]` holds an array of `WeightFeedingEntry` rows — one per life stage ("3 Months" / "6 Months" both-sex rows, "Adult" split male/female) — each with a `weightKgRange`, `calorieRange`, and `kibbleGramsRange`. `ADULT_THRESHOLD_YEARS[breed]` is the age (1/1.5/2/3 yrs, per the breed's real maturity) at which a pet's guide switches from "6 Months" to "Adult" figures. `weightFeedingStage(ageYears, breed)` buckets a plain `ageYears` decimal into one of the three stages; `weightFeedingEntry(pet)` resolves the matching row (falling back to the stage's "both" row when `pet.sex` is unset); `weightTargetRange(pet)` (replaced the old flat `WEIGHT_TARGETS` map) returns just the `weightKgRange` for the pet-detail weight-chart band. `dailyGramTarget` now sources its kibble-grams estimate from this guide (midpoint of `kibbleGramsRange`) ahead of any static per-breed fallback. `CARE_PLANS[breed].items` no longer carry per-breed gram numbers in `perDayGrams` — those numbers live in the guide table instead, since they vary by age/sex; item `detail` text folds in each breed's real emergency callouts (bloat/GDV, brachycephalic heatstroke, Scottish Fold progressive arthritis watch-signs, Golden Retriever cancer self-check, etc.) from the reference doc. `/plan` and `/pet/[id]` both render a small "Weight & feeding guide" card from `weightFeedingEntry(pet)` above/near the existing content, labeled with the pet's current age/stage so it's clear the numbers move as the pet ages. (`pet.sex` IS editable post-creation now — Settings ▸ Family edit-pet sheet, Male/Female/Not-set segmented — so the old known-gap is closed; a pet with no sex on file still gets the stage's neutral "both" row.)
  - **Custom (off-list) breeds:** the Add-a-pet breed field (`/pets`) is a free-text input with a type-to-filter suggestion dropdown (only shows once the user types; matches `BREEDS_BY_SPECIES` case-insensitively and saves the canonical spelling). A breed with no `CARE_PLANS` entry is a "custom breed": on `/plan` it gets a fully **editable** care plan instead of the vet-built one — the daily numeric targets (`CUSTOM_TARGET_FIELDS`, edited via `EditStatSheet` → `pet.customPlan.{fedPerDay,fedGrams,waterPerDay,litterPerDay/walkPerDay}`) **plus** a "Grooming, health & vet" section of non-daily activities (`OTHER_CARE_FIELDS` in `app/plan/page.tsx` — grooming, nails, dental, weight check, parasite, vet checkup, meds, etc.) whose frequencies are editable free-text via `EditTextSheet`, stored in the generic `pet.customPlan.cadences[id]` map (falls back to each field's default cadence). `customPlan` persists as the `custom_plan` JSON column, so new sub-keys need no migration.
- `lib/store.tsx` — the **Supabase-backed single source of truth**: React context + all actions (`logAction`, `buyCosmetic`, `toggleEquip`, reminders CRUD, `addPet`/`editPet`/`deletePet`, members CRUD, premium, `switchMember`, `restockSupply`/`useSupply`, `bookVetById`, `setUnits`, `setSeenWelcome`, `setNotificationPref`, `setFamilyPassword`, `signOut`). Key internals: `createClient()` (one browser client), **optimistic writes reconciled by `persist()`** (rolls back the touched slice + toasts "That change didn't save — reverted" only on a real error), **`undoableDelete()`** (soft-delete + Undo toast, ~5s grace) for reminder/med deletes, **`computeStreak(activities)`** (streak from real local-calendar history), **debounced `syncCounters()`** (single coins/streak write — avoids the rapid-tap DB write-ordering race; `rewardsRef` holds the authoritative running coins total), and the notification simulation (`notifyRecentActivity`/`notifyCareWarnings`, deduped **per recipient** via `${memberId}:${id}` keys — note `notifyCareWarnings` no longer raises the "N care warnings" summary toast; the bell badge is the sole surface for outstanding-alert counts). `bootstrapHousehold()` is the fallback seed when the signup trigger didn't create a household. **Treat store logic as stable; most UI changes shouldn't need to touch it.**
  - **Health/scheduling store additions:** `addVaccination` (sequenced insert — vaccination row first, then its linked reminder, because of the FK), `deleteVaccination` (undoable; locally removes/restores the linked reminder, DB cascades it), `addVetVisit`/`deleteVetVisit` (visit records do NOT log an activity — `/logs`' vet tap pairs `logAction` with an optional details sheet), `addWeight(petId, kg, ts?)` (client-generated `id`, sorted insert, headline `weightKg` tracks the latest point) + `deleteWeight` (undoable, refuses the last point, keeps `weightKg` == latest), `logAction(petId, type, grams?, ts?)` (retro logging — a non-today `ts` skips ALL same-day alert raise/resolve machinery), `undoLogAction` (compensating delete: −5 coins via `rewardsRef`, streak recomputed, supply drain refilled; wired as the Undo action on every log toast), and `editPet`'s widened patch (`sex`/`birthDate`/`microchip`/`allergies`/`notes` are **presence-guarded** — only keys present in the patch are written, so narrow callers can't wipe them; `birthDate` in the patch also writes derived `age_years`). `toggleReminder` on a recurring reminder rolls `due` forward via `nextRepeatDue` instead of completing (alert reminders never carry `repeatKind`, so alert machinery is untouched). Load() runs a **vaccine-due catch-up** (nextDue within 30 days / ≤7 days overdue with no reminder carrying that `vaccinationId` → creates one; note: deleting that reminder while still in the window re-creates it next load — by design, "the shot is still due"). **Pre-migration fallback:** if the `vaccinations`/`vet_visits` embed errors (migrations 0013/0014 not applied), hydration retries with `LEGACY_HOUSEHOLD_SELECT` and sets `legacySchemaRef` — reminder inserts then omit the v2 columns; health features stay inert but the app loads.
- `lib/supabase/` — `client.ts` (browser client), `server.ts` (SSR/cookie client), `middleware.ts` (`updateSession` — the auth wall + session refresh; early-returns when env is missing), `env.ts` (`supabaseEnvReady` gate). `lib/authErrors.ts` — `friendlyAuthError()` humanizes raw Supabase auth strings on login/signup.
- `proxy.ts` (repo root) — the Next middleware entry: redirects signed-out users to `/login` and signed-in users away from auth routes.
- `supabase/migrations/` — forward-only SQL: schema, RLS policies, the `handle_new_user` trigger (seeds a new household), and later fixes (e.g. `0009_seed_cat_weight.sql` corrects the seed cat's weight). **Never edit a shipped migration; add a new one.** Latest: `0013_health_records.sql` (pets `birth_date`/`microchip`/`allergies`/`notes`, `vaccinations` + `vet_visits` tables with membership RLS, `reminders.vaccination_id`) and `0014_reminders_v2.sql` (`reminders.repeat_kind`/`repeat_interval`). **These must be applied in the Supabase dashboard before the health features go live** — until then the store's legacy-select fallback keeps the app running with those features inert.

- **Accessibility prefs are device-local** (`lib/a11y.tsx`): Reduce Motion / Reduce Transparency are stored in **localStorage** via `useSyncExternalStore` (hydration-safe, no setState-in-effect) and reflected as `data-reduce-motion` / `data-reduce-transparency` attributes on `<html>`, which `globals.css` keys overrides off. This is a deliberate, narrow exception to the "no localStorage" rule — display/a11y prefs belong to the *device*, not the shared household row, so they need **no migration** and don't touch the Supabase store. `AccessibilityProvider` is mounted in `PhoneShell`.
- Pets are tappable everywhere (Home hero, Settings ▸ Family list) → `/pet/[id]`. **Home pet carousel**: with >1 pet, the Home hero switches pets on **horizontal swipe** (>45px horizontal, ~1.4× more horizontal than vertical, so it ignores taps/vertical scroll); a `didSwipe` ref suppresses the hero `<Link>` when the gesture was a swipe.
- **Favicon** is Mozart (the classic seed cat): `app/favicon.ico` is the 16×16 `CAT_SPRITE` from `petSprites.ts` rendered to a multi-size ICO (a small Pillow `NEAREST`-upscale script regenerates it).

## Constraints / gotchas
- **Phone-only**: always render as a phone. On desktop it sits inside an iPhone-style bezel with a dynamic island; on real phones it's full-screen. Don't build desktop-wide layouts.
- **Real backend now**: Supabase auth + RLS. Every route is behind the `/login` wall (`proxy.ts`); to reach the app in a browser you need an account (ask the user for a test login). RLS scopes all data to the signed-in user's household. Payments/booking are still faked (instant premium unlock, "booking requested").
- **Env**: needs `NEXT_PUBLIC_SUPABASE_URL` + `NEXT_PUBLIC_SUPABASE_ANON_KEY` in `.env.local` (copy `.env.example`). Without them the app shows a `MissingEnv` config screen instead of crashing (the middleware skips the auth wall so the page can render it). `.env.local` is gitignored; `.env.example` is committed.
- Environment is Windows. Both Bash (Git Bash) and PowerShell tools are available. **Headless Chrome IS available** at `C:\Program Files\Google\Chrome\Application\chrome.exe` — screenshot the running app (`--headless=new --screenshot=... http://localhost:PORT/`) and drive it via the DevTools protocol (`--remote-debugging-port`, then a small WebSocket client / the `scratchpad/cdp.js` driver). Gotchas: `--screenshot` only writes reliably over `http://` (a `file://` page hits "Access is denied"); use a **fresh `--user-data-dir` per run** (Chrome caches fonts across headless runs by `@font-face` family); you'll hit the `/login` wall first, so log in (set the React inputs via the native value setter + dispatch an `input` event, then submit) or reuse an authed profile. Fast Refresh can serve a stale component tree — hard-reload before testing edits. `fonts.gstatic.com` font binaries fail from Git Bash `curl` but work via PowerShell `Invoke-WebRequest`.
- The Vercel MCP server needs interactive auth; a non-interactive session can't run the OAuth flow. Deploy via the `vercel` CLI or ask the user to authorize.

## What has been tested / how to verify
- Baseline each change with `npm run build` + `npx tsc --noEmit` + `npm run lint` (all must be clean).
- Smoke test after `npm run dev` / `npx next start`: log in, then confirm all routes render (`/`, `/activity`, `/plan`, `/reminders`, `/pets`, `/vets`, `/settings` + sub-pages, `/pet/<uuid>`, `/pet/<uuid>/card`, `/join?f=<uuid>`). Signed out, everything redirects to `/login`. (Signup has email confirmation ON — a fresh throwaway can't log itself in; use the shared test account.)
- Full manual flow at a mobile viewport (~390×844): log a care action (toast + feed entry + coins + `+5` pop; CoinPill bumps in the shop sheet), switch family member, add/complete/**undo-delete** a reminder, buy + equip a cosmetic (pet preview updates, hat pokes above the frame), toggle PetPal+ (care plan unlocks, sponsored vet suggestion + booking appear, dashboard checklist shows), request a vet booking, switch weight units (editors + chart both follow) — **reload between checks to prove the DB, not just local state.**
- The store has been hardened across five remediation passes (correctness/forgiveness, feedback/animation, WYSIWYG, a11y/consistency, robustness/hygiene). When touching store logic, preserve: optimistic-write rollback, undo-able deletes, computed streak, debounced counter writes, and per-recipient notification dedupe.
- Keep the IDE Problems panel clean. Tailwind "canonical class" (`suggestCanonicalClasses`) warnings: use the canonical utility the IDE suggests — arbitrary → scale step (`h-[6px]`→`h-1.5`, `h-[42px]`→`h-10.5`, `h-[52px]`→`h-13`, `w-[220px]`→`max-w-55`, `aspect-[2/1]`→`aspect-2/1`), `[scrollbar-width:none]`→`scrollbar-none`, `bg-gradient-to-*`→`bg-linear-to-*`, and move a leading `!` to a **trailing** `!` (`!h-[42px]`→`h-10.5!`). **Watch out:** the IDE sometimes suggests an invalid font-size class like `text-15px` — that utility doesn't exist and silently drops the style, so for arbitrary font sizes keep `text-[15px]` and only fix the `!` position. After any such rewrite, verify the computed px is unchanged (headless Chrome + DevTools can read `getComputedStyle`).
- ESLint must be clean. Notable react-hooks rules in play: don't write a ref's `.current` during render (do it in an effect), and don't `setState` synchronously in an effect — for "read an external system on mount" cases (e.g. a media query) use a lazy `useState` initializer + subscribe-only effect. The one-time Supabase hydration in `store.tsx` is an intentional exception with a scoped `eslint-disable-next-line`.
